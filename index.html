This specification defines the syntax and semantics of a behavior definition language. A service may send instructions encoded in this format for clients to realize. These messages include instructions about positioning objects, activating or deactivating systems, changing scenes or contexts, controlling visual properties, animating characters, and playing audio.

Introduction

This specification aims to establish a messaging protocol that allows for finer execution control of verbal and nonverbal agent systems.

Scope

This section is informative.

This document is limited to specifying the syntax and semantics for a behavior definition language. One example application for Act messages would be a mobile device application that requires careful manipulation, execution, and synchronization of Behavior's such as movements and speech.

Goals and Motivation

This section is informative.

An Act message is JSON language for controlling the verbal and nonverbal behavior of conversational agents. An Act Message specifies the execution of Behavior objects and any underlying synchronization details.

Message Format

This section is formative.

Messages Interface

            typedef sequence< Act > actSequence;
            interface Messages{
                attribute actSequence acts;
            };

acts attribute

In defining the data type for the acts attribute we defined a custom typedef such that a Sequence of Act's corresponds to an actSequence. The sequence data type is defined in [[WebIDL-20161215]]. ] The acts attribute then denotes a sequence containing any number of Act messages. 

                [
                    Act1,
                    Act2,
                    Act3,
                    ...
                ]

Act Interface

            typedef sequence< Block > blockSequence;
            interface Act{
                attribute double version;
                attribute long id;
                attribute blockSequence blocks;
            };

version attribute

The version attribute specifies the release of the Act message format to be used by a double numeric data type. 

                1.2

id attribute

The id attribute specifies the unique identifying number belonging to the corresponding Act message. 

                0

blocks attribute

In defining the type for the blocks attribute we defined a custom typedef such that a sequence of Block objects corresponds to a blockSequence. The blocks attribute then specifies a list of Block's. See Block below. 

                [
                    Block1,
                    Block2,
                    Block3,
                    ...
                ]

Block Interface

            typedef sequence< Behavior > behaviorSequence;
            interface Block{
                attribute behaviorSequence behaviors;
            };

behaviors attribute

In defining the type for the behaviors attribute we defined a custom typedef such that a sequence of Behavior's corresponds to behaviorSequence. The Sequence data type is defined in [[WebIDL-20161215]]. The behaviors attribute then contains any number of the Behavior objects. See Behavior below. 

                [
                    Behavior1,
                    Behavior2,
                    Behavior3,
                    ...
                ]

Behavior Interface

The atomic unit that can be immediately realized by a client is called a behavior. Consider the sentence, "The boy says hello." By analyzing the grammar, we find the subject is "boy", the verb is "says", and the direct-object is "hello". The intuition behind the behavior interface is to represent a similar structure. 
            typedef sequence< String> stringSequence;
            interface Behavior {
                attribute long id;
                attribute String object;
                attribute String action;
                attribute double delay;
                attribute Params param;
                attribute Dictionary start;
                attribute stringSequence closedCaption;
            };

id attribbute

The id specifies the unique identifier within the containing block. Note that id's are non unique across different Block's.

object attribute

The object attribute is the name of the agent that is performing an action. In grammar, this is often called the subject of a verb. Examples include: 

              "boy"

And even abstract nouns such as the following. 

              "system"

action attribute

Each object affords a set of actions. The action attribute specifies one of those available actions. For example, if the object is "boy", a possible action may look as follows. 

              "say"

And if the object is "system", a possible action may be an abstract concept as follows. 

              "initialize"

delay attribute

The delay attribute specifies the number of seconds to wait to perform a corresponding action through a double data type. 

              0.0

param attribute

The Params attribute specifies the data-structure containing all data to be received and manipulated by the client. See Params below.

start attribute

The start attribute is an optional Dictionary argument that specifies when the Behavior should begin its execution. It is specified by two key:value pairs where the first pair is denoted by a "trigger":String pairing, while the second pairing is denoted by a "id":long pairing. The long data type is defined in [[WebIDL-20161215]].

trigger key

The trigger attribute specifies the point at which the Behavior corresponding to the specified id should begin its execution. 

                    "start"
                
                    "end"

id key

The id key specifies the Behavior that will be affected by the corresponding trigger value. 

                0
            
                {
                    "trigger" : "end",
                    "id":0
                }

closedCaption attribute

In defining the closedCaption attribute, a custom typedef was used such that a sequence of the String's corresponds to a ClosedCaption data type. The String data type defined in [[WebIDL-20161215]] is used. The closedCaption attribute is then specified by a list of DCMP standard abiding String's, containing the textual representation of any corresponding events within the message.

The following example describes the closed captioning for the event where a character named Frog excitedly says 'Hi there!'. 

              [
                  "[Frog excited]",
                  "Hi there!"
              ]

The following example describes the closed captioning for the event where a character named Pig says 'Welcome back!' in a Southern Accent. 

              [
                  "[Pig Southern Accent]",
                  "Welcome back!"
              ]

The following example shows the closed captioning for the event where three characters are simultaneously cheering. 

              "[Frog, Pig, Goose cheering]"

The following example shows the closed captioning for the event where a Cow is Mooing. 

              "[Cow moo's]"

Params Interface

            typedef Dictionary Phonemes;
            interface Params {
                attribute String intent;
                attribute String SSML;
                attribute String name;
                attribute String url;
                attribute Phonemes phonemes;
                attribute String language;
                attribute Dictionary context;
            };

intent attribute

The intent attribute specifies what a user is trying to accomplish. An intent is specified within a string data type. 

                "statement.i-can-help"

SSML attribute

The SSML attribute specifies the speech synthesis markup language for the specified intent. The SSML is specified within a string data type. 

                "I can help you with that!"

name attribute

The name attribute specifies the name of the audio file to be played. 

                "GO_0020"

url attribute

The url attribute specifies the source location of the audio file to be played. The url attribute is specified within a string data type. 

                "http://audioURL..."

phonemes attribute

The phonemes attribute specifies a Dictionary containing the phonetic translation and total number of frames belonging to the corresponding SSML text. See Phonemes below. 

                {
                    "segments": [
                                    {
                                        "phonemeLabel": "IY",
                                        "startFrame": 3
                                    }
                                ],
                    "framecount": 202
                }

language attribute

The language attribute specifies the tongue in which any text, or audio will be communicated in. The language is specified by the three letter codec abiding ISO-639. The following example shows the three letter codec for the English language. 

              "eng"

context attribute

The context attribute specifies a Dictionary as defined in [[WebIDL-20161215]]. The dictionary members are specified in key:value pairs where both key and value of of type String. 

                {
                    "category": "colors",
                    "gamestate" : "1",
                    "target" : "blue"
                }

Phonemes Interface

            typedef sequence< Dictionary > dictionarySequence;
            interface Phonemes {
                attribute dictionarySequence segments;
                attribute long framecount;
          };

segments attribute

A custom typedef was defined such that a sequence of Dictionary's corresponds to a dictionarySequence. Each Dictionary object within the segments attribute is a Dictionary containing a pair of key:value attributes. The first item in each nested Dictionary is specified by a String:String pairing denoting a phonetic label. The second Dictionary item corresponds to a String : double pairing corresponding to the start frame of the phonetic label. 

                {
                    "phonemeLabel": "IY",
                    "startFrame": 3
                }

framecount attribute

The framecount attribute specifies the total number of feature frames of the transcribed SSML text. 

                202

Examples

This section is normative.

The following are examples of objects and their associated behaviors. Note that the specified parameters for each of the following actions are normative.

Animate

An animate payload specifies an object to be animated by the client. 

              {
                "object": "goose",
                "action": "animate",
                "delay": 0,
                "params": { "name": "FR_Maracas_1" }
              }

Say

A say payload specifies what the desired object should say.

              {
                "object": "frog",
                "action": "say",
                "delay": 0,
                "params": {
                  "intent": "statement.i-can-help",
                  "SSML": "I can help you with that!",               // utterance previously
                  "name": "GO_0020",                                 // audio file name
                  "url": "http://audioURL...",
                  "phonemes":{
                                  "segments": [
                                                  {
                                                      "phonemeLabel": "IY",
                                                      "startFrame": 3
                                                  }
                                              ],
                                  "framecount": 202
                             }
              }

Do

A do payload specifies the name of the action the specified target should perform.

              {
                "object": "goose",
                "action": "do",
                "delay": 0,
                "params": { "name": "walk", "context" : ["pig"]}
              }

Play

A play payload specifies the audio clip the client should play.

              {
                "object": "sound",
                "action": "play",
                "delay": 0,
                "params": { "name": "sxf.bonusround.wav" }
              }

Spawn

A spawn payload notifies the client to display the specified object.

              {
                "object": "system",
                "action": "spawn",
                "delay": 0,
                "params": {"name": "frog", "context": ["character2"]}
              }

Destroy

A destroy payload notifies the client to remove the specified object.

              {
                "object": "system",
                "action": "destroy",
                "delay": 0,
                "params": {"name": "frog"}
              }

Timing Control

This section is informative.

In order to synchronize the behaviors(especially the "say" and "animate" actions), we propose to use the "message block". A message block is an ordered list of Behavior objects, where each object conforms to the Behavior message format specified within this document. The server sends sequences of blocks, and the client (e.g. Unity) must fully process one block of Behavior before reading the next Block. While processing a single block of Behavior, the client should schedule Behavior by putting them on "tracks"(every action of every object has a track), then all the tracks start simultaneously.

This section is informative.

In the following example we examine events A and B: we want to enforce that event B can't start until event A has started.

Right when Goose starts speaking, Fox jumps.

              "object": "goose",
              "action": "say",
              "params": {"name": "GO_0020" , "intent": "statement.i-can-help", "SSML": "I can help you with that!"}

              "action": "animate",
              "params": {"name": "FX_Jump"}
            }

In this example we want to run the statements in a particular sequence. Specifically, B can't start until A has finished. In order to achieve the desired order, we place each behavior in a separate block.

            {
              "object": "goose",
              "action": "say",

            }
          ]

          [
            {
              "object": "fox",
              "action": "animate"
              "params": {"name": "FX_Jump"}
            }
          ]

Related Technologies

This section is informative.

BML vs. Act Blocks

The following BML snippet simultaneously starts an animation and a speech utterance. 

        <bml character="Alice">
            <pointing target="blueBox" mode="RIGHT_HAND" start="speech1:start"/>
            <speech id="speech1">
                <text>Look there!</text>
            </speech>
        </bml>

The equivalent behavior can be represented in an Act Messages as follows.

      [
        {"object": "Alice", "action": "do", "params": {"name": "point", "context": ["blueBox", "mode": "RIGHT_HAND"]}},
        {"object": "Alice", "action": "say", "params": {"intent": "look"}}
      ]

BML documentation recommends using <wait> to align behavior with a condition or an event.

      <bml character="Alice">
        <gesture id=”g1” type=”point” target=”object1”/>
        <body id=”b1” posture=”sit”/>
        <wait id=”w1” condition=”g1:end AND b1:end”/>
        <gaze target=”object2” start=”w1:end”/>
      </bml>

The <wait> is unnecessary since we can synchronize behaviors by placing them in different blocks.

      [
        {"object": "Alice", "action": "do", "params": {"name": "point", "context": ["object1"]}},
        {"object": "Alice", "action": "do", "params": {"name": "body", "context": ["sit"]}}
      ]

      [
        {"object": "Alice", "action": "animate", "params": {"name": "gaze"}}
      ]

Multi-party behavior synchronization is limited. It's non-trivial to have two characters say something at the same time as BML.

      <bml character="Alice">
          <speech><text>Yay!</text></speech>
      </bml>

      <bml character="Bob">
          <speech><text>Yay!</text></speech>
      </bml>

Act message format allows fine-control of multi-party behaviors, with a natural syntax.

      [
        {"object": "Alice", "action": "say", "params": {"intent": "yay"}},
        {"object": "Bob", "action": "say", "params": {"intent": "yay"}}
      ]
